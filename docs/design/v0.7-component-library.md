# v0.7 — Component library

**Status:** Planned. Acceptance criteria: `AGENTS.md` §5.3.

## Goal

Reusable, parametric building blocks — FuelPin, GuideTube, ControlRod,
ReflectorBlock, WaterBox — plus hexagonal-lattice support, so a user
(or the v0.8 generator) composes "a FuelPin with these dimensions"
instead of hand-wiring surfaces.

## Design

### Architecture

```text
src/promptmc/components/
├── __init__.py        # COMPONENT_REGISTRY
├── fuel_pin.py        # FuelPin(params) -> Universe (pellet/gap/clad/moderator)
├── guide_tube.py
├── control_rod.py
├── reflector_block.py
└── water_box.py
```

Each component is a **parametric factory**: a small Pydantic params
model in, a `GeometryModel` fragment (typically a sub-`Universe`) out —
composable into lattices and assemblies.

```python
class FuelPinParams(BaseModel):
    pellet_radius: float = Field(gt=0)
    clad_inner_radius: float = Field(gt=0)
    clad_outer_radius: float = Field(gt=0)
    pitch: float = Field(gt=0)
    fuel_material_id: int
    clad_material_id: int
    moderator_material_id: int
    # model_validator: pellet < clad_inner < clad_outer < pitch/2
```

### Key decisions

- **Parameter validation at the boundary.** Nested-radius ordering
  (pellet < clad inner < clad outer < half-pitch) is the classic bug;
  it is a `model_validator`, not a runtime surprise.
- **Boundary type is a parameter, not an assumption.** A single pin is
  reflective (infinite lattice); a finite assembly may be vacuum.
  Components never hard-code the choice.
- **Hex lattices extend the schema**: a `HexLattice` node in
  `geometry/primitives.py` mapping to `openmc.HexLattice` in the
  serializer, alongside the existing rect-lattice path.
- **Pre-approved means gated.** Every component must pass the v0.6
  physics gate before it ships. The gate, not review alone, is the
  quality bar.
- **No gold-plating.** Five components + hex lattices is the scope;
  speculative components wait until a benchmark or user needs them.

## Testing

- Per component: valid params → fragment that passes the v0.6 gate and
  round-trips through v0.3 serialization; invalid params (pellet ≥
  clad) → Pydantic `ValidationError`.
- A small hex assembly of FuelPins builds, passes the gate, and
  round-trips.

## Open questions

- Material handling: components reference material IDs — should the
  library also ship standard material definitions (UO₂, Zircaloy,
  water at standard conditions), or stay geometry-only and leave
  materials to the caller?
- Are components exposed as MCP tools in this release, or
  library-only until the v0.8 generator consumes them?
