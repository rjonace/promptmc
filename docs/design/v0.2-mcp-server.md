# v0.2 — MCP server

**Status:** Shipped (v0.2.0 2026-05-30, refined through v0.2.2)

## Goal

Expose PromptMC's validation and execution logic to AI assistants
(Claude Desktop/Code, Cursor, VS Code, …) via the Model Context
Protocol, so an AI client can drive an end-to-end OpenMC workflow
through schema-validated tools instead of writing ad-hoc Python.

## Background

AI assistants are good at proposing OpenMC configurations and bad at
guaranteeing they are valid. Rather than let a model write XML
directly, the MCP server puts every AI interaction behind typed tools
whose inputs and outputs are Pydantic schemas — the same validation
surface the CLI uses.

## Design

### Key decisions

- **The MCP layer parallels the CLI; it does not extend it.** Every
  tool wraps existing library functionality. No new behavior hides
  behind the MCP surface, and `cli.py` is never modified for a tool.
- **PromptMC is not the agent.** The server provides tools to *some
  other* agent. No LLM calls inside the MCP layer — provider choice
  belongs to the client driving it.
- **Three-layer separation:**
  - `mcp/schemas.py` — Pydantic input/output schema per tool. Use
    `Literal` for constrained strings (e.g. severity) so the generated
    JSON Schema constrains LLM clients.
  - `mcp/tools.py` — pure functions, no SDK dependency, unit-testable
    in isolation.
  - `mcp/server.py` — stdio wiring only; imports the MCP SDK lazily so
    plain CLI use pays no import cost.
- **Synchronous tools only.** No streaming, no `openmc_run_async`;
  async orchestration is the client's concern.
- **Trusted local invocation.** stdio transport, no auth layer. The
  server runs with the user's own privileges on their own machine.
- **Results return as text content** (JSON in `TextContent`) rather
  than structured content — settled in v0.2.1 after MCP SDK validation
  errors with `outputSchema` (see CHANGELOG).

### Tool surface (10 tools)

| Tool | Wraps |
|---|---|
| `openmc_check_installation` | `OpenMCInstaller.check_installation()` |
| `openmc_validate` | `OpenMCValidator.validate_input_file()` |
| `openmc_schema_check` | `SchemaValidator.validate_directory()` |
| `openmc_template` | `get_template().render()` |
| `openmc_run` | `OpenMCRunner.run_simulation()` |
| `openmc_analyze` | `ResultParser.parse_results()` |
| `openmc_list_templates` | `list_templates()` |
| `openmc_check_cross_sections` | cross-section discovery |
| `openmc_plot` | OpenMC native 2D plotting → base64 PNG |
| `openmc_geometry_debug` | OpenMC `--geometry-debug` |

Plus three resources: `promptmc://cross-sections`,
`promptmc://history`, `promptmc://examples/uo2_criticality`.

`openmc_plot` is the visual-verification answer: 2D slice plots
rendered natively into the chat client, instead of a web UI (which is
explicitly out of scope, per ROADMAP constraints).

### Follow-up decisions (v0.2.2)

- `mcp` promoted from optional extra to core dependency — the server is
  the product surface, not an add-on.
- The opt-in LLM planner consolidated on Google Gemini behind a thin
  `GeminiClient` seam with structured outputs (`GEMINI_API_KEY` /
  `GEMINI_MODEL`), replacing the OpenAI-compatible configuration.

## Testing

- Every tool has a unit test that exercises it **through the MCP
  layer**, not just the underlying function.
- An integration test spawns the server in a subprocess and round-trips
  `tools/list` and `tools/call`.
- A dedicated `mcp-tests` CI job runs the suite with coverage.

## Constraints

- No LLM calls in the MCP layer (hard invariant).
- Core library remains importable without OpenMC; only execution tools
  require it.
