# v0.5 — Geometry composition + inspection

**Status:** Planned. Acceptance criteria: `AGENTS.md` §5.3.

## Goal

Deterministic MCP tooling on top of the v0.3 schema and v0.4 library:
build a validated geometry from semantic JSON, and inspect existing
geometries through a lean query surface. No physics judgment, no LLM.

## Design

### New tools

- **`openmc_build_geometry`** — input: a `GeometryModel` as JSON plus
  an output path; output: the written XML path and a structural summary
  (cell/surface counts, bounding box). Validation then serialization,
  nothing else.
- **`openmc_query_geometry`**, **`openmc_list_cells`**,
  **`openmc_list_materials`** — read a `GeometryModel` (or parse an
  existing XML) and return structured summaries: counts, names, IDs,
  fills.

### Key decisions

- **Mirror the v0.2 tool pattern exactly.** Pydantic input/output
  schemas in `mcp/schemas.py`, pure functions in `mcp/tools.py`,
  registration in `mcp/server.py`. This release is intentionally
  low-novelty; the value is the surface, not the mechanism.
- **Errors are structured results, not exceptions.** An invalid model
  returns a validation report across the MCP boundary; it does not
  raise.
- **ID handling matches v0.3 auto-assign rules**, so build → inspect
  round-trips agree on IDs.
- **Keep inspection lean.** Three tools, not a kitchen sink.
  `openmc_trace_point` belongs to the physics gate (v0.6) and
  `openmc_diff_geometry` to generation (v0.8) — they are not built
  here.

```python
# mcp/tools.py (shape)
def openmc_build_geometry(inp: BuildGeometryInput) -> BuildGeometryOutput:
    model = GeometryModel.model_validate(inp.geometry)   # schema gate
    serialize(model, inp.output_path)                    # v0.3 path
    return BuildGeometryOutput(
        output_path=str(inp.output_path),
        summary=_summarize(model),
    )
```

## Testing

- Each tool unit-tested **through the MCP layer**, plus entries in the
  integration round-trip (the v0.2 precedent).
- Valid model → XML written, summary correct. Invalid model →
  structured error, no exception across the boundary.
- v0.4 benchmarks reused as inspection fixtures.

## Open questions

- Input ergonomics: accept only the full `GeometryModel` JSON, or also
  a path to an existing geometry XML for the inspection tools? (Current
  lean: both for inspection, JSON-only for build.)
- Should the structural summary include a computed bounding box when
  OpenMC is absent, or only when it is available to compute one
  reliably?
