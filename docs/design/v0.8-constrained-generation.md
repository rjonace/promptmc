# v0.8 — Constrained generation

**Status:** Planned. Acceptance criteria: `AGENTS.md` §5.3.

## Goal

Natural language → validated, runnable OpenMC input. The capstone the
earlier releases exist to make safe: the LLM emits a `GeometryModel`
(constrained by the v0.3 JSON Schema), which is structurally validated
and then gated by the v0.6 physics checks before anything runs.

## Design

### Architecture

```text
src/promptmc/generation/
├── __init__.py
├── llm_clients.py   # Generator protocol + GeminiGenerator + MockGenerator
├── pipeline.py      # prompt -> generate -> validate -> repair -> serialize
└── repair.py        # turn v0.6 ValidationIssues into a corrective follow-up prompt
```

### The validate-and-repair loop

```python
def design(description: str, gen: Generator, max_repairs: int = 3):
    schema = GeometryModel.model_json_schema()
    raw = gen.generate(build_prompt(description), schema)
    for _ in range(max_repairs + 1):
        try:
            model = GeometryModel.model_validate_json(raw)   # schema gate
        except ValidationError as e:
            raw = gen.generate(repair_prompt(description, raw, e), schema)
            continue
        issues = validate(model)                             # v0.6 physics gate
        if not any(i.severity == "error" for i in issues):
            return model                                     # human still reviews
        raw = gen.generate(repair_prompt(description, raw, issues), schema)
    raise GenerationError("did not converge to a valid model")
```

The repair prompt is built from the gate's structured
`ValidationIssue`s — their `fix_hint`s are why the gate's failure
explanations had to be machine-readable.

### Key decisions

- **`Generator` protocol behind a thin internal interface.** Google
  Gemini (structured output against the model's JSON Schema) is the
  supported provider; a deterministic `MockGenerator` serves tests.
  Adding a provider later is an addition, not a rewrite. User-facing
  provider freedom lives in the client-agnostic MCP server, not here.
- **Bounded repair, structured failure.** The loop either passes the
  gate or exits with the last validation errors. No unbounded retries.
- **Never trust LLM output.** Schema + physics validation is mandatory;
  unknown nuclides are rejected; prompt content never bypasses the
  gates.
- **The agent is not the product.** Generation is exposed as an
  explicit tool (`openmc_design(description: str)`); PromptMC still
  provides tools to some other agent rather than acting autonomously.
- **`promptmc plan` is rebuilt on the pipeline** and the old keyword
  router deleted — CLI surface unchanged, implementation replaced.
- **`openmc_diff_geometry(before, after)`** shows a reviewer exactly
  what the repair loop changed between iterations.
- **Minimal audit record** (model used + artifact produced) ships here;
  the full provenance trail is v0.9.
- **Human-in-the-loop is a hard line.** `design()` returns a
  *candidate*; a person approves before it is treated as real. Never
  autonomous for licensing or safety-critical decisions.

## Testing

- **No LLM calls in tests.** `MockGenerator` returns prerecorded JSON;
  a few real outputs are captured once, manually, and frozen as
  fixtures.
- Pipeline logic: first-invalid-then-valid repair succeeds;
  never-valid raises after `max_repairs` with the final issues
  attached.
- The real Gemini client stays thin and is not unit-tested against the
  live API.

## Open questions

- Repair budget: is `max_repairs = 3` right, and should it be
  user-configurable through the MCP tool input?
- Should `openmc_design` also emit materials/settings, or geometry
  only with templates covering the rest? (Scope of "validated input
  package" needs pinning.)
- How much of the v0.7 component library should the prompt encourage —
  free-form CSG, components-first, or a tiered prompt that prefers
  components and falls back to primitives?
