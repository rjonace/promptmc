# v0.4 — Reference geometry library

**Status:** Planned. Acceptance criteria: `AGENTS.md` §5.3.

## Goal

~6 validated reference geometries built as typed `GeometryModel`s on
the v0.3 schema, each runnable, documented, and proven against
published benchmark results. This release also stabilizes the schema
(the experimental flag drops).

## Background

In reactor physics, credibility comes from reproducing published
benchmarks. A library whose Godiva actually runs to k ≈ 1.0 against
ICSBEP is both the trust artifact for the project and the regression
suite every later release (physics gate, components, generation) is
tested against.

## Design

### The library

```text
src/promptmc/benchmarks/
├── __init__.py        # ALL_BENCHMARKS registry (name -> module)
├── godiva.py          # ICSBEP HEU-MET-FAST-001 (exists, v0.3)
├── jezebel.py         # ICSBEP PU-MET-FAST-001
├── pwr_pin.py         # Mosteller PWR pin cell (exists, v0.3)
├── bwr_pin.py         # BWR pin cell
├── icsbep_<case>.py   # one or two additional fast cases
└── _types.py          # Benchmark protocol (exists, v0.3)
```

Every module exposes the same tiny surface — `NAME`, `SOURCE`
(citation), `EXPECTED_KEFF`, `KEFF_TOLERANCE`, `build()` — so the test
harness and a future `promptmc examples` command can iterate them
uniformly. Selection keeps at least one shielding/streaming case and
one fast/criticality case alongside the thermal-reactor geometries.

### Key decisions

- **Specs come from the published sources** (ICSBEP handbook, Mosteller
  reports), not from secondary summaries. Each module docstring cites
  the source and records the expected k-eff, tolerance, and modeling
  notes.
- **Materials are half the work.** A correctly-shaped geometry with the
  wrong enrichment or density will not be critical; densities and
  compositions are cross-checked against the benchmark spec.
- **Schema stabilization is a one-way door.** Any reshaping the
  benchmarks force on the v0.3 schema happens here, before the
  experimental flag drops; afterwards the schema is public API across
  minors.

### The headline test

```python
@pytest.mark.requires_openmc_data
@pytest.mark.parametrize("bench", ALL_BENCHMARKS)
def test_keff_matches_reference(bench, tmp_path):
    # build -> serialize -> run OpenMC -> StatePoint.keff
    assert abs(k - bench.EXPECTED_KEFF) < 3 * sigma
```

It needs cross-section data, so it cannot run in hosted CI: it is
gated behind `requires_openmc_data` and run locally/nightly. CI keeps
the structural layer (every benchmark builds, serializes, and has the
intended object graph) so the library stays covered without OpenMC.

### `promptmc compare` (verification surface)

The headline test proves k-eff against the reference, but it lives in the
test suite. `promptmc compare <benchmark> <statepoint>` exposes the same
check as a command: it reads the computed k-eff from a statepoint,
compares it to the benchmark's `EXPECTED_KEFF` within `KEFF_TOLERANCE`,
and reports the deviation (absolute and in σ) with a pass/fail and a
non-zero exit on failure. `--json` follows the structured-output
convention so CI and agents can consume it. This is what turns
"independently checked against known results" into a reproducibility
artifact a user (or a JOSS reviewer) reruns in one command.

Two rules keep it honest. It shares a single tolerance policy with the
headline test (the open question below), so "pass" means one thing in
both places. And it is read-only over a statepoint a prior `promptmc run`
produced; it does not run OpenMC, so it stays fast, deterministic, and
usable without cross-section data.

### Benchmarks as MCP resources

Per the "MCP Parallels CLI" constraint, the library is also exposed on
the MCP side as read-only resources (e.g. `promptmc://benchmarks/godiva`),
one per entry in `ALL_BENCHMARKS`, alongside the existing single example
resource. Each returns the benchmark's serialized geometry and materials
plus its citation and expected k-eff, so an assistant can read a
known-good geometry as grounding and as a few-shot seed for v0.8
generation. The registry stays the single source of truth: the same
`ALL_BENCHMARKS` backs the headline test, `promptmc compare`,
`promptmc examples`, and these resources, and `build()` stays OpenMC-free
so resources work from a bare install.

## Testing

- Structural build/serialize tests for all benchmarks in CI.
- k-eff round-trip suite locally with real cross sections, tuned so σ
  is small enough that a correct model passes and a wrong one fails.
- Round-trip (model → XML → re-parse) for every geometry.

## Out of scope

- Pre-run validators (v0.6) and the `openmc_build_geometry` tool
  (v0.5). v0.4 is only the validated library.

## Open questions

- Which ICSBEP fast cases round out the six? Selection should align
  naming and expected values with the OpenMC community's benchmark
  conventions.
- Does the BWR pin need lattice support that the v0.3 schema lacks, or
  can it ship as a single bounded cell like the PWR pin?
- Tolerance policy: 3σ statistical only, or σ combined with the
  benchmark's reported experimental uncertainty?
