# v0.1 — CLI initial release

**Status:** Shipped (v0.1.0 2026-05-24, refined through v0.1.4)

## Goal

A production-grade CLI wrapper around OpenMC: configure, validate, run,
and analyze simulations without hand-editing XML or reading HDF5, plus a
deterministic natural-language planner for getting started.

## Background

OpenMC's native workflow is XML inputs, manual batch management, and
HDF5 outputs. The friction is highest for newcomers (which XML files,
which settings?) and for iteration (re-running sweeps, parsing
statepoints). v0.1 wraps that workflow behind a typed, validated CLI.

## Design

### Key decisions

- **Typer + Rich for the CLI.** Typed command signatures, automatic
  help, rich terminal output. Commands live one-per-module under
  `commands/`; `cli.py` only registers them (split in v0.1.4 after the
  monolith reached ~730 lines).
- **Dual execution modes.** `OpenMCRunner` drives OpenMC via the Python
  API when importable and falls back to subprocess invocation otherwise
  (`ExecutionMode`: auto/api/subprocess). The core library must import
  and validate *without* OpenMC installed — only execution requires it.
- **Three focused integration classes.** The original heavyweight
  `OpenMCIntegration` class was split (v0.1.2) into `OpenMCInstaller`
  (detection), `OpenMCValidator` (input checking), and `OpenMCRunner`
  (execution) — single responsibility, independently testable.
- **Pydantic at the boundary only.** XML schema validation
  (`schema.py`, via `defusedxml`) uses Pydantic models for
  `settings.xml` / `materials.xml` / `geometry.xml`; internal value
  objects are dataclasses.
- **Deterministic planner first.** `promptmc plan` (originally `ask`)
  maps plain-English requests onto configuration templates with a
  keyword router — no generative AI in the default path, so behavior is
  reproducible. LLM planning is opt-in (`--llm`).
- **Templates as the unit of configuration.** Criticality, fixed
  source, shielding, and reactor-pin templates (`templates.py`) render
  validated `settings.xml` rather than asking users to write XML.
- **Telemetry optional.** OpenTelemetry lives behind the `telemetry`
  extra and is imported only inside `telemetry.py`; absent, everything
  no-ops.

### What was tried and removed

- **Plugin system** (v0.1.0) — a registry with entry-point discovery
  and lifecycle hooks. Removed in v0.1.1: speculative surface area with
  no users. Deleting it everywhere in one commit set the project's
  "no shim half-states" refactoring norm.
- **Separate `parallel.py`/`performance.py` modules** — merged into
  `batch.py`/`progress.py` (v0.1.1) once it was clear they were one
  concern each.

## Architecture

```text
src/promptmc/
├── cli.py                 # app setup + command registration only
├── commands/              # plan, run, validate, analyze, batch, templates, info, configure
├── openmc_integration.py  # OpenMCInstaller / OpenMCValidator / OpenMCRunner
├── schema.py              # Pydantic XML validation (defusedxml)
├── templates.py           # config templates + registry
├── assistant.py           # deterministic NL planner (+ optional LLM)
├── batch.py               # batch sweeps + parallel execution
├── resources.py           # resource limits, temp workspaces
├── progress.py            # progress reporting, system profiling
├── visualization.py       # statepoint/tally parsing (StatePoint.keff)
├── errors.py              # PromptMCError hierarchy, logging, retry
└── telemetry.py           # optional OpenTelemetry
```

## Testing

- Unit tests per module; CLI tested through Typer's runner with mocked
  integration classes.
- Shipped at 218 tests / >80% coverage; the coverage-must-not-decrease
  rule dates from this release.
- CI: ruff (lint + format), mypy strict, pytest, bandit.

## Constraints

- Python 3.10+, 80-char lines, strict typing — set here and unchanged
  since.
- `shlex.join` for shell construction; `defusedxml` for parsing.
