# v0.9 — Observability, provenance + audit

**Status:** Planned. Acceptance criteria: `AGENTS.md` §5.3.

## Goal

Opt-in OpenTelemetry instrumentation across the MCP tool surface, plus
a deterministic local record of exactly what an AI agent did:
per-tool-call spans, usage metrics, and an append-only `audit.jsonl`
with client/model attribution.

## Posture (non-negotiable)

This is the release where it is easiest to violate the project's own
rules, so they come first:

- **Opt-in and off by default.** No exporter configured → strictly
  no-ops, asserted by tests.
- **Local.** It traces the operator's own tool calls for their own
  inspection (or their own infrastructure's audit). It is **not**
  product analytics, and PromptMC never transmits anything.
- **Isolated.** `opentelemetry` is imported only inside `telemetry.py`
  (guarded by `_OTEL_AVAILABLE`), behind the `telemetry` extra.

## Design

### Tracing and metrics

Extend the existing `telemetry.py` with a decorator applied at the MCP
dispatch layer, so every tool is covered uniformly:

```python
def traced_tool(name: str):
    """Span + counters per tool call. No-op when telemetry is disabled."""
```

Metrics: tool-call volume, payload size, schema-rejection rate.
Exporter and endpoint come from standard `OTEL_*` environment
variables; absent → disabled.

### Provenance and the audit sink

- **Client capture (automatic):** the MCP `initialize` request carries
  `clientInfo`; its name is attached to every span/record as
  `llm_product`.
- **Model/provider capture (env injection):** MCP does not pass the
  model (the client app owns API keys and routing), so it is read from
  env vars the client sets in its server config:
  `PROMPTMC_TRACKING_MODEL` → `llm_model`,
  `PROMPTMC_COMPANY_ID` → `llm_company`. Missing vars degrade to
  `"unknown"`, never crash.
- **The audit sink:** one JSON object per tool call appended to a local
  `audit.jsonl` — tool name, arguments, outcome, timestamp, and the
  three attribution dimensions:

```json
{"ts": "...", "tool": "openmc_design", "args": {...}, "ok": true,
 "llm_product": "Claude Desktop", "llm_model": "gemini-2.5-pro",
 "llm_company": "Google"}
```

Together these let an operator answer, on their own machine: which
client and model drove which tool calls, and how many were rejected by
validation — debugging for the agent loop, not an analytics pipeline.
An operator may ship `audit.jsonl` to their own SIEM; PromptMC itself
never uploads it.

## Testing

- Telemetry disabled (default): tools behave identically and import
  cleanly with OpenTelemetry not installed. Off-by-default is a test
  case, not a comment.
- In-memory span exporter (test-only): one tool call → one span with
  expected attributes, counter incremented.
- Audit enabled: exactly one well-formed `audit.jsonl` line per call;
  disabled: no file written.

## Open questions

- Audit file location and rotation: fixed path in the working
  directory, configurable path, or per-session files? Append-only
  forever is simple but unbounded.
- Should tool *arguments* be recorded verbatim (paths may be
  sensitive even locally) or summarized/hashed with an opt-in for full
  capture?
- Does enabling audit require the `telemetry` extra, or should the
  `audit.jsonl` sink work without OpenTelemetry installed (plain JSON
  writes), keeping only spans/metrics behind the extra?
